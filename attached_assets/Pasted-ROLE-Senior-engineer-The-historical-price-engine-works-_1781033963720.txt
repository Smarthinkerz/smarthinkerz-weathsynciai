ROLE

uses RAW (unadjusted) close prices. Raw prices create phantom single-day returns on
split/dividend dates (a 4-for-1 split looks like a -75% day), which silently corrupts
volatility, max drawdown, beta, and Sharpe for any stock with a corporate action in the
window. Add a guard so these metrics can never be silently wrong, and confirm the
adjusted-data upgrade path. Keep every existing honesty guarantee.

HARD CONSTRAINTS
- Engine stays LLM-free; the no-AI guard test must still pass. No AI produces any number.
- Never silently emit a corrupted number. When the data basis can't support a trustworthy
  metric, say so — same philosophy as "insufficient data."
- Provenance labels stay accurate (raw vs adjusted must be truthful).

TASK 1 — CORPORATE-ACTION JUMP GUARD (for the raw-price path)
In the risk engine, before computing volatility/drawdown/beta/Sharpe, scan the daily-return
series for any single-day move beyond a high threshold (e.g. |return| > 35%) — the signature
of an unadjusted split or large special dividend, not normal price action.
  - If such a jump is detected AND the price basis is "raw_close": do NOT silently compute as
    if it were a real return. Mark the affected metrics as "unreliable — possible unadjusted
    split/dividend in window; upgrade to adjusted prices for accurate stats," and surface that
    note in the API response and the UI panel.
  - Do NOT auto-delete the day blindly (a real -35% crash is legitimate) — the honest move is
    to FLAG the metric as potentially unreliable, not to fabricate a clean number.
  - Make the threshold a named constant with a comment explaining the split rationale.

  - the UI basis note reads "split/dividend adjusted" instead of the raw-close caveat.
The basis (raw vs adjusted) must drive the labeling and the warning automatically.
